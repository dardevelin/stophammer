# ADR 0018: Proof-of-Possession for Authorized Feed and Track Mutations

## Status
Accepted (Phase 1 implemented; see Implementation Status below)

## Context

Stophammer is a public index — any crawler can submit a feed, and feeds are indexed by
their `podcast:guid`. There is no account system. The node has no concept of "this artist
owns this feed."

This creates a gap: there is currently no mechanism for the legitimate owner of a feed to:

- Remove a feed from the index (`FeedRetired`)
- Remove a specific track (`TrackRemoved`)
- Relocate a feed to a new RSS URL
- Relocate a track's audio to a new enclosure URL

The `FeedRetired` and `TrackRemoved` event types already exist in the event log and are
propagated to community nodes, but `apply.rs` does not implement them — community nodes
log and skip both. There is also no HTTP endpoint that emits either event. Deletion has
never worked end-to-end.

### Why identity is hard here

Stophammer identifies feeds by `podcast:guid`, which is generated by the hosting platform
(e.g. Wavlake, RSS Blue, Buzzsprout) — not by the artist. An artist has no session, no
account, and no pre-registered key in stophammer.

The only artifacts the system stores that belong to the artist are:

- **Payment route addresses** (`payment_routes.address`, `feed_payment_routes.address`) —
  the Lightning node pubkey or Lightning address configured in their RSS feed. For
  self-custody wallet users this is a key they control; for custodial platform users
  (e.g. some Wavlake accounts) this key belongs to the platform, not the artist.
- **Audio files** — the artist controls the bits in the enclosure, even if hosted elsewhere.
- **RSS feed content** — the artist controls what goes into the RSS on most platforms.

### The problem with simpler approaches

**"Just call a DELETE endpoint"** — anyone who knows a `feed_guid` could delete it.
No authorization exists to gate the operation.

**"Require an API key"** — there is no account system and we do not want to build one.
Artists should not need to register with stophammer to manage their own content.

**"Use the payment address as identity, require a keysend"** — works for self-custody
wallet holders. Fails completely for custodial platform users whose payment address is
owned by the platform, not the artist. Also requires stophammer to run a Lightning node,
which is infrastructure not yet in scope.

**"Use stophammer.json at a well-known URL"** — only works for artists who already have
a web presence and a prior trust anchor. A fresh `stophammer.json` with no prior
relationship to a feed can be published by anyone — there is no bootstrapping protection.
This is the right long-term enrichment path but cannot serve as the identity anchor for
mutations.

## Prior art

### RFC 8555 — ACME (Let's Encrypt)

RFC 8555 is the closest formal precedent. ACME solves the same class of problem:
"prove you control a domain/resource without a pre-existing account, then receive a
credential scoped to that resource." Its pattern:

1. Client requests a **challenge** for an identifier (domain name)
2. Server issues a token; client embeds it at a well-known location (HTTP path or DNS
   TXT record)
3. Server fetches the location, verifies the token is present, transitions the
   authorization object to `valid`
4. Client receives a credential (certificate) scoped to that identifier

ACME's state machine for challenge objects is `pending → processing → valid | invalid`.
We adopt this explicitly.

ACME's token format (§8.1) binds the token to the requester's account key:

```
keyAuthorization = token || '.' || base64url(SHA-256(accountPublicKey))
```

This prevents replay: a third party who *observes* the token in the RSS cannot reuse it
because the server checks that the token is bound to the key that originally requested
the challenge.

We adopt the same binding. In our case there is no pre-registered account key, so we
bind instead to a **requester nonce** — a fresh random value generated client-side and
included in the `POST /proofs/challenge` request. The stored token is:

```
token || '.' || base64url(SHA-256(requester_nonce))
```

The `POST /proofs/assert` request must include the original `requester_nonce`. The server
recomputes the binding and rejects any assertion that does not match.

RFC 8555 §11.3 also mandates ≥128 bits of entropy in the server-issued token. We follow
this requirement.

### RFC 7235 + RFC 6749 + WebAuthn

- The two-step pattern name — **challenge** then **assert** — is confirmed by RFC 7235
  (HTTP authentication challenges) and WebAuthn
  (`AuthenticatorAssertionResponse`). No better-established alternative name was found.
- The issued credential is an **access token** (RFC 6749), scoped via **scope** claims.
- The overall mechanism is **proof-of-possession** (RFC 7800).

No standard was found that names this entire pattern as a single coined term. ACME,
domain verification (Let's Encrypt, GitHub, Google), and WebAuthn all describe the same
structure but do not give it a unified name. We use "proof-of-possession" as the umbrella
term throughout this codebase.

## Decision

### Vocabulary

| Term | Meaning here | Source |
|------|-------------|--------|
| **challenge** | Token the server issues that the requester must embed somewhere only they can write | RFC 7235, WebAuthn, RFC 8555 |
| **assert** | Presenting evidence that the challenge was embedded correctly | WebAuthn (`AuthenticatorAssertionResponse`) |
| **proof-of-possession** | The overall mechanism: prove control of a resource by completing a challenge | RFC 7800, RFC 8555 |
| **access token** | Short-lived bearer token issued after a successful assertion, scoped to one resource | RFC 6749 |
| **scope** | What the token permits: `feed:write` | RFC 6749, RFC 8693 |
| **pending / valid / invalid** | Challenge object states | RFC 8555 §7.1.6 |

**Rejected names:**
- `ownership` — vague, does not map to a REST resource
- `auth` — implies a general account system, which does not exist
- `retire` / `relocate` in URLs — invented verbs; standard HTTP verbs (`DELETE`, `PATCH`)
  carry the mutations instead

`/proofs` names the resource: a proof-of-possession flow in progress or completed.

### API design

```
POST /proofs/challenge
  Request:  { feed_guid, scope, requester_nonce }
  Response: 201 { challenge_id, token, state: "pending", expires_at }

POST /proofs/assert
  Request:  { challenge_id, requester_nonce }
  Response: 200 { access_token, scope, subject_feed_guid, expires_at }
           or 4xx { error, state: "invalid", reason }

with Authorization: Bearer <access_token>:

DELETE /feeds/{guid}
PATCH  /feeds/{guid}                    body: { feed_url }
DELETE /feeds/{guid}/tracks/{track_guid}
PATCH  /tracks/{guid}                   body: { enclosure_url }
```

Standard HTTP verbs on existing resources carry the mutations. No invented verbs appear
in URLs. `DELETE` means remove from index. `PATCH` means update a mutable field.

### Challenge state machine (from RFC 8555 §7.1.6)

```
POST /proofs/challenge
       │
       ▼
   [ pending ]   ← token issued, not yet verified
       │
       │  POST /proofs/assert → server fetches RSS + audio, looks for token
       ├──────────────────────────────────────────────────────────────┐
       ▼                                                              ▼
   [ valid ]                                                    [ invalid ]
  access_token issued                                      reason returned, challenge consumed
```

Challenges do not retry — a failed assertion transitions to `invalid` and is consumed.
The requester must call `POST /proofs/challenge` again to start over.

### Token format and security (RFC 8555 §8.1, §11.3)

- Server-issued token: 128 bits of entropy, base64url-encoded (~22 characters)
- Stored token binding: `token || '.' || base64url(SHA-256(requester_nonce))`
- The `requester_nonce` is generated by the client, sent in both the challenge request
  and the assert request. The server never stores the nonce — only the binding.
- This prevents a passive observer of the RSS from replaying the token against a
  different requester.
- Challenge expires after **24 hours**. Access token expires after **1 hour**.
- Both are single-use. Consumed on assertion (success or failure).

### Challenge embedding (proof of RSS + audio control)

To assert, the owner must embed the full token binding string in **two places**:

1. **The RSS feed** — in a `<podcast:txt>` tag at channel level. The parser reads this
   on the next crawl.
2. **The audio file at the enclosure URL** — as an ID3 `TXXX` frame (`description` =
   `stophammer-proof`, `value` = token binding), a FLAC `STOPHAMMER-PROOF` comment, or
   as ASCII text within the first 4 KB of the file (stophammer scans only the header).

Requiring both proves:
- The requester writes to the RSS (controls the hosting account).
- The requester controls the audio file location (can modify or re-upload it).

Neither alone is sufficient: an RSS mirror cannot modify the audio; a CDN operator
cannot modify the RSS.

For **relocations** (`PATCH /feeds/{guid}` or `PATCH /tracks/{guid}`), the token must
appear at **both the old and the new location**. This prevents URL squatting: you cannot
claim a new URL without proving you still control the old one.

### What changes in the event log

No new event types are introduced. This ADR provides the authorization gate that allows
the existing `FeedRetired` and `TrackRemoved` events to be emitted. Implementing those
handlers in `apply.rs` (currently stubbed) is a separate concern.

### What this does not solve: platform-custodial artists

Artists whose RSS is platform-generated and whose audio lives on platform CDN cannot
complete the two-part embedding — they have write access to neither.

**Wavlake** is the identified example. Some Wavlake artists use Wavlake's custodial
wallet and cannot edit the raw RSS XML or modify files on Wavlake's CDN. These artists
need a platform-specific path: a Wavlake export tool, a Wavlake transfer API, or
cooperation from Wavlake. This is a partnership or tooling problem, not a protocol
problem.

This is explicitly out of scope. Platform-custodial migration must not drive the design
of the general flow, and may require a separate project entirely.

## Consequences

- Any artist who controls their RSS feed and audio files gets a working delete and
  relocate path with no account registration required and no Lightning node dependency.
- The proof mechanism is transport-independent: plain HTTPS, no keysend, no pre-existing
  stophammer relationship required.
- The token binding (adopted from RFC 8555) prevents passive replay attacks by observers
  of the RSS feed.
- `FeedRetired` and `TrackRemoved` can now be fully implemented in `apply.rs`: cascade
  delete from `feeds`/`tracks`, `payment_routes`, `value_time_splits`, `entity_quality`,
  and the FTS5 `search_index`. The `delete_from_search_index` function already exists in
  `search.rs` but is currently uncalled.
- Platform-custodial artists (Wavlake and similar) are a documented, deferred gap.
- The `access_token` exists only at the HTTP layer of the primary node. It is not an
  event, not a signing key, and does not replicate to community nodes.
- Two new tables are required: `proof_challenges` (pending/valid/invalid challenges) and
  `proof_tokens` (issued access tokens with scope and expiry). Names chosen over the
  previously considered `ownership_challenges` / `access_tokens` to match the `/proofs`
  resource name consistently.

## Implementation Status

### Phase 1 (implemented)

- **RSS `podcast:txt` verification**: The `verify_podcast_txt` function fetches the RSS
  feed and checks for a `<podcast:txt>` element containing the token binding string at
  channel level. This is the sole proof mechanism currently in production.
- **Token revocation on `feed_url` change**: When `PATCH /feeds/{guid}` updates the
  `feed_url`, all `proof_tokens` for that `feed_guid` are revoked atomically within the
  same transaction. The artist must re-prove ownership against the new URL. This prevents
  a token proved against an old URL from authorizing mutations after a URL change.
- **ACME-inspired challenge/assert flow**: Full two-step flow with requester nonce
  binding, challenge expiry (24h), and access token expiry (1h).
- **SSRF validation**: `validate_feed_url` rejects private/reserved IPs and non-HTTP
  schemes before any RSS fetch.
- **Assertion race detection**: The `POST /proofs/assert` handler has a three-phase
  structure: phase 1 reads the `feed_url` under the DB lock, phase 2 performs async RSS
  verification against that URL, and phase 3 re-acquires the lock to issue the token. A
  concurrent `PATCH /feeds/{guid}` could change the `feed_url` between phases 1 and 3,
  meaning the RSS verification was performed against a URL that is no longer current.
  Phase 3 now re-reads the feed and compares the current `feed_url` to the one used in
  phase 2. If they differ, the handler returns HTTP 409 (Conflict) with the message
  "feed URL changed during verification; retry" and does not issue a token. This follows
  RFC 9110 Section 15.5.10 (409 Conflict): the request could not be completed due to a
  conflict with the current state of the target resource. The client should re-start the
  challenge/assert flow from `POST /proofs/challenge`.

### Phase 2 (not implemented)

The following capabilities are described in the Decision section above but are **not yet
implemented**. They are tracked as future work.

- **Audio file proof-of-possession**: The Decision section describes embedding the token
  binding in an ID3 `TXXX` frame, a FLAC `STOPHAMMER-PROOF` comment, or as ASCII in the
  first 4 KB of the audio file. This second proof layer (proving control of the audio,
  not just the RSS) is not implemented. The current system trusts RSS control alone as
  sufficient proof.
- **Dual-location verification for relocations**: The Decision section specifies that
  `PATCH /feeds/{guid}` and `PATCH /tracks/{guid}` should require the token to appear at
  both the old and new URL. This is not implemented. Instead, the current implementation
  revokes all tokens after a `feed_url` change, requiring a fresh proof against the new
  URL. This is a weaker but operationally simpler guarantee.
- ~~**`FeedRetired` / `TrackRemoved` in `apply.rs`**~~: Implemented. `apply.rs` now
  handles both event types with full cascade hard-deletes (feed/track rows, child payment
  routes, value time splits, and FTS5 search index entries). Community nodes apply these
  events identically to the primary.
